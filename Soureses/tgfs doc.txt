                  DOKUMENTACJA ARCHITEKTURY I STRUKTURY DANYCH
                       TGFS (Tag Graphic File System) v1.0
================================================================================
Status specyfikacji: Produkcyjna / Sterownik Niskopoziomowy x86_64
Zasada działania:    Płaska struktura bezfolderowa, indeksacja tagami bitowymi
Złożoność:           O(1) – Selekcja zasobów realizowana wewnątrz rejestrów CPU
--------------------------------------------------------------------------------

1. LOGICZNA STRUKTURA SEKTORÓW DYSKU (LBA MAP)
----------------------------------------------
System plików TGFS porzuca tradycyjne, głębokie i powolne struktury drzewiaste 
na rzecz płaskiej przestrzeni indeksowej (Tag Registry). Układ fizyczny sektorów 
wygląda następująco:

  [SEKTOR 0] -> MASTER BOOT RECORD (MBR) / GUID PARTITION TABLE (GPT)
                Zarezerwowane dla kodu startowego maszyny i bootloadera.

  [SEKTOR 1] -> TGFS SUPERBLOCK
                Główny sektor identyfikacyjny systemu plików.
                * Bajty 0..3:   Sygnatura magiczna (Ciąg ASCII: "TGFS")
                * Bajty 4..7:   Wersja systemu plików (Zwykle 0x00000001)
                * Bajty 8..15:  64-bitowy wskaźnik LBA wskazujący lokalizację 
                                rejestru tagów (Tag Registry LBA).

  [SEKTOR R] -> TAG REGISTRY (Rejestr indeksów i deskryptorów plików)
                Sektor ten zawiera tablicę struktur opisujących zasoby. Pojedynczy 
                sektor (512 bajtów) mieści dokładnie 8 rekordów po 64 bajty każdy.

  [SEKTOR D] -> OSTRZESZEŃ I EXTENTY DANYCH (Raw Data Blocks)
                Ciągłe, niepofragmentowane bloki zawierające surowy kod programów 
                oraz grafiki interfejsu GUI zaalokowane sekwencyjnie.


2. SPECYFIKACJA REKORDU PLIKU (64-BYTE FILE DESCRIPTOR STRUCT)
--------------------------------------------------------------
Każdy plik zarejestrowany w systemie TGFS opisany jest sztywną strukturą 
zajmującą dokładnie 64 bajty wewnątrz Tag Registry:

  - Przesunięcie 0  (4 bajty):  Unikalne ID pliku (File ID, 32-bit Integer).
                                Wartość 0 oznacza rekord pusty/skasowany.
  - Przesunięcie 4  (8 bajtów): 64-bitowa maska tagów (1 bit = 1 cecha pliku).
  - Przesunięcie 12 (8 bajtów): 64-bitowy wskaźnik pierwszego sektora LBA pliku.
  - Przesunięcie 20 (8 bajtów): Całkowity rozmiar pliku w bajtach.
  - Przesunięcie 28 (36 bajtów): Nazwa pliku ASCII (Dla czytelności / celów GUI).


3. DEFINICJA SPECYFIKACJI TAGÓW SYSTEMOWYCH
--------------------------------------------
Filtrowanie cech odbywa się poprzez operacje na maskach bitowych (Bitmask). 
Podstawowe stałe architektury to:

  - Bit 0 (0x01): TAG_SYSTEM      -> Pliki jądra systemu, tablice IDT/GDT, fonty.
  - Bit 1 (0x02): TAG_GUI         -> Zasoby interfejsu graficznego (ikony, motywy).
  - Bit 2 (0x04): TAG_APPLICATION -> Pliki wykonywalne aplikacji i programów.
  - Bit 3 (0x08): TAG_IMAGE       -> Surowe bitmapy graficzne TrueColor (XRGB).


4. ANALIZA PORÓWNAWCZA PRĘDKOŚCI (TGFS vs NTFS/EXT4)
----------------------------------------------------
  A. Wyszukiwanie zasobów GUI:
     - Klasyczny OS: Otwiera folder tekstowy -> Czyta nazwy -> Porównuje ciągi 
       znaków ("icon.png") znak po znaku. Operacje tekstowe obciążają CPU.
     - Twój TGFS: Pobiera maskę `TAG_GUI | TAG_IMAGE`. Wykonuje sprzętowy rozkaz 
       `and rax, r12` bezpośrednio w rejestrze procesora. Wybór następuje 
       w ułamku nanosekundy.

  B. Odczyt dużych plików (Brak fragmentacji):
     - Pliki w TGFS alokowane są wyłącznie w sposób ciągły (Extents). Podczas 
       wywołania `tgfs_load_file_by_id`, system nie musi skakać po łańcuchach 
       klastrów w tablicach alokacji. Wysyłana jest jedna komenda DMA do 
       kontrolera SATA AHCI, przerzucająca całą aplikację bezpośrednio do 
       pamięci RAM z maksymalną prędkością elektroniki dysku SSD.
